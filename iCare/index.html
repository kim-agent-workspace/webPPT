Architecture Overview

系统总体框架

客户端与前端
系统以 Web 浏览器与 Electron 桌面端作为双入口，前端基于 Vue 3 构建慢病管理工作台，并结合 Vue Router 与 Pinia 实现页面路由与状态管理。
服务层与数据层
后端采用 ThinkPHP 8.1 多应用架构，配合 Kafka 数据平台接入多源异构数据；业务数据存储于 MySQL，卫健相关数据与算法支撑数据存储于 PostgreSQL，共同形成慢病全周期数据基础。
RPA 与 AI 预测
针对社服系统等封闭外部系统，平台通过 Playwright 自动化补足数据抓取与回填能力；在此基础上，结合事件—特征变化驱动分桶与离散时间生存模型，实现慢病风险预测与辅助决策。
图：iCare 系统总体框架 · 客户端 → 前端 → 后端 → RPA → 外部系统 → 数据层 → AI 预测层

Section 1.2

框架图说明

三条逻辑主线：数据流 + 业务流 + 自动化流。

Flow 01 · 数据流
多源接入 → 统一存储
社服系统、HIS、LIS、公卫系统等多源异构数据通过 Kafka 数据平台接入，经解析、清洗、字段映射、身份匹配、时间戳统一和质量校验后，写入 MySQL 主业务库，与 PostgreSQL 共同形成慢病全周期数据基础。
Flow 02 · 业务流
Web + 桌面端双入口
前端基于 Vue 3 构建慢病管理工作台，后端 ThinkPHP 8.1 通过 Controller 完成业务管理、接口服务和公共能力封装，桌面端通过 Electron + IPC 通信提供本地化能力。
Flow 03 · 自动化流
RPA 补足 + AI 预测
通过 RPA 自动化层模拟人工操作，解决社服系统等封闭系统的数据互通问题。AI 预测层基于患者全周期数据，采用事件-特征变化驱动分桶法和离散时间生存模型，预测疾病进展、并发症、转诊等多种结局风险。
核心判断：数据结构决定慢病 AI 模型是否可训练、可解释、可追溯。

Section 2.3

三表式数据库设计

围绕患者慢病全周期管理，设置三类核心实体表，形成完整、连续、可追溯的数据链条。

Table 01
患者主索引表

作用：全库患者身份识别与跨机构数据关联的基础，实现同一患者在不同机构、不同系统、不同时间段内数据的统一归集。

核心字段：患者唯一标识、基本人口学信息、疾病类型、所属医疗机构、转入转出转诊记录。

模型意义：为模型训练提供唯一患者身份锚点，确保跨源数据可关联。

Table 02
随访队列表

作用：记录随访期间仍持续产生管理记录的患者，以及结局尚未明确、仍处于随访或待核实状态的患者。

核心字段：队列状态、随访开始时间、最近有效随访时间、检查评估时间、持续管理标志、待核实标志。

模型意义：动态管理随访窗口，明确适用于风险预测和预警算法的人群范围。

Table 03
终点事件队列表

作用：记录已明确发生终点事件的患者，定义死亡、严重并发症、急性加重、住院、靶器官损害、疾病进展、转诊、失访、迁出等结局。

核心字段：终点事件类型、发生时间、确认来源、事件说明、相关证据。

模型意义：为模型训练提供清晰、可追溯的标签定义，明确患者结局状态。

RPA 背景 · Background
13 / 25
Section 3.1

建设背景：为什么需要 RPA

核心痛点

社服系统等第三方异源机构系统不开放标准接口，无法通过以下常规方式完成数据对接：

  • REST API
  • 数据库直连
  • HL7 / FHIR 标准协议
本质上不是"破解接口"，而是在合规账号权限范围内，通过自动化浏览器代替人工重复操作。

技术链路

Electron 桌面端 — 整体架构容器
Vue 3 渲染进程 — 发起 RPA 任务请求
Node.js 主进程 — 任务调度与执行管理
Playwright + Chromium — 控制内置浏览器访问社服系统

功能介绍：ExtJS 适配 · 可靠性 · 安全

ExtJS 页面适配

  • 层级定位 + 文本匹配 + 原生 DOM click
  • 等待遮罩层消失后再操作
"高保真模拟输入"策略
模拟真实用户的点击、全选、删除、逐字符输入、Tab 失焦等动作，确保社服系统正确触发表单校验和数据保存逻辑。

可靠性设计

  • 全局互斥锁：同一时间只运行一个 RPA 任务
  • 登录状态检查：任务开始前检查浏览器状态和登录状态
  • 自动恢复：登录失效自动重新登录；浏览器崩溃自动重建实例
  • 异常分类处理：账号错误、验证码失败、患者无记录、网络超时、浏览器崩溃、ExtJS 遮罩卡死等场景分类处理

安全与部署

  • 凭据通过 electron-store 加密存储，不在源码中硬编码
  • RPA 操作运行在隔离的 Chromium 实例中
  • 数据访问和 OCR 服务均部署在内网环境，通信不经过公网
  • 浏览器内置打包，无需用户安装额外浏览器

离散时间生存模型：问题定义

传统离散时间比例风险模型要求协变量影响满足比例风险结构。机器学习离散时间生存模型放宽这个限制，直接学习个体在每个时间区间内的条件事件风险。

Hazard 定义

\[ h_{ij} = P(T_i \leq t_{ij} \mid T_i > t_{i,j-1}, x_{i,j-1}) \]
\[ h_{ij} = f_\theta(x_{i,j-1}, t_{i,j-1}, t_{ij}) \]

ti,j−1

hij

离散时间生存模型：损失函数

对每个个体、每个时间区间，模型输出 hij = fθ( xi,j−1 , ti,j−1 , tij )。事件区间似然贡献为 hij，非事件区间为 1 − hij。整体条件似然取负对数：

\[ \mathcal{L}(\theta) = -\sum_{i=1}^{n}\sum_{j=1}^{k_i}\left[Y_{ij}\log h_{ij} + (1-Y_{ij})\log(1-h_{ij})\right] \]

符号说明

  • n：个体数量
  • ki：第 i 个个体的观察区间数量
  • Yij：事件标签（0 或 1）
  • hij：模型预测的区间事件概率
该 Loss 本质上是二分类似然在生存时间区间上的扩展。

两种情况的训练推力

事件区间：Yij = 1
损失项 = −log hij，模型必须把 hij 预测得越高越好
非事件区间：Yij = 0
损失项 = −log(1 − hij)，模型必须把 hij 预测得越低越好

→ 在真正发生事件的区间给出高风险，在未发生事件的区间给出低风险。

医保费用预测、资源调配与智能门诊随访

基于门诊量、挂号记录、就诊人次、疾病谱、年龄结构和历史医疗支出数据形成预测能力。

门诊峰谷预测

  • 按小时或分时段统计和预测患者到诊人数
  • 根据挂号数据形成计数型分析结果
  • 识别患者集中到诊的时间峰谷和空间分布差异
  • 患者端：提供次日就诊建议
  • 管理端：提供未来一周至一个月的门诊峰谷预测

医疗资源调配

  • 刻画就诊需求与医疗资源供给之间的错位
  • 针对基层医疗机构不分专科、资源利用不均衡等情况
  • 辅助医院进行门诊排班、随访安排和资源调配
  • 面向卫健管理端提供区域级资源规划依据

医保费用预测与监测

  • 结合就诊人数、疾病结构和历史费用数据
  • 支撑医保费用预测与医保支出监测
  • 可面向患者端、医院端、卫健管理端提供不同决策支持
  • 支持区域医保基金支出趋势分析与预警
